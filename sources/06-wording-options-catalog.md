# Catalog of AI-Attribution Wording Options

*Scope: enumerate and evaluate the full space of phrasings actually used or proposed to credit AI in commits — real-world usage, git-trailer parseability, connotation, DCO/legal safety, GitHub contribution-graph effect, and best-fit context. Feeds the final recommendation matrix.*

Research date: 2026-07-08.

> **Sourcing note.** Each row is anchored to a primary source: the actual tool/project policy text, git's reference manual, foundation legal pages, or the U.S. Copyright Office guidance. "Real-world" = a shipping tool emits it or a project policy prescribes it; "proposal" = it appears only in a spec thread, blog, or issue. Confidence is flagged per claim and summarized at the end. Cross-refs: §01 (trailer mechanics), §02 (DCO/sign-off).

---

## 0. Two facts that govern every row

**(a) Any hyphenated `Token: value` line is a valid, machine-parseable git trailer.** Git's trailer machinery parses RFC-822-style `token: value` lines in a blank-line-separated block at the end of the message; the token may contain hyphens but **not spaces**, and any token can be registered/queried via `trailer.<token>.*` config and `git log --grep`/`--format=%(trailers)`. So `Co-authored-by`, `Assisted-by`, `Generated-by`, `Co-developed-by`, `Aided-by`, `Prompted-by`, `Tool` are *all* equally well-formed trailers to git — git assigns none of them special meaning. ([git-interpret-trailers](https://git-scm.com/docs/git-interpret-trailers), [git-config](https://git-scm.com/docs/git-config)) The one non-trailer option is a plain body-text note (no `token:` line, or with an internal space like `Made with AI`), which git treats as ordinary prose. *Confidence: high.*

**(b) The contribution-graph effect is identical for every trailer and is driven only by email, not by the token.** GitHub links a commit/co-author to an account **only** by matching the email to a registered account (or that user's `noreply`). A synthetic AI address (`noreply@anthropic.com`, `copilot@github.com`) matches no account, so **no phrasing inflates any human's graph**, and `Co-authored-by` is the *only* token GitHub renders specially (as a co-author chip) — the others are inert log text to GitHub. See §01 §3–4. ([GitHub Docs: multiple authors](https://docs.github.com/en/pull-requests/committing-changes-to-your-project/creating-and-editing-commits/creating-a-commit-with-multiple-authors), [why commits link to the wrong user](https://docs.github.com/articles/why-are-my-commits-linked-to-the-wrong-user)) *Confidence: high.* **Consequence:** the "GitHub contribution-graph effect" column below is essentially constant — the real differentiators are *connotation*, *DCO/legal safety*, and *whether GitHub renders a co-author chip*.

**Legal anchor for the connotation column.** The U.S. Copyright Office guidance states: *"to qualify as a work of 'authorship' a work must be created by a human being"* and *"Applicants should not list an AI technology or the company that provided it as an author or co-author simply because they used it when creating their work."* ([USCO, 88 FR 16190, Mar. 16 2023, p.3 §II & p.5 §IV.A](https://www.copyright.gov/ai/ai_policy_guidance.pdf)) This is why "co-author"/"author" wording carries a legal-authorship connotation that the office affirmatively warns against, and why every AI-permitting foundation policy insists the *human* is the sole author and the AI is a *tool*.

## 1. The catalog (one row per phrasing)

Columns: **Parseable trailer?** (well-formed git trailer), **GitHub co-author chip?** (does GitHub render it as a linked co-author), **Connotation**, **DCO/legal safety** (does it risk implying AI legal authorship / conflict with human sign-off).

### Real-world usage (a tool emits it or a policy prescribes it)

| Phrasing | Real-world usage / who uses it | Parseable trailer? | GitHub co-author chip? | Connotation | DCO / legal safety | Best-fit context |
|---|---|---|---|---|---|---|
| **`Co-Authored-By:`** | **The de-facto tool default.** Claude Code default (`Co-Authored-By: Claude <noreply@anthropic.com>`, plus a `🤖 Generated with Claude Code` line) ([Claude Code issue #66602](https://github.com/anthropics/claude-code/issues/66602), [settings docs](https://code.claude.com/docs/en/settings)); VS Code Copilot (`Co-authored-by: Copilot <copilot@github.com>`, setting `git.addAICoAuthor`) ([VS Code #314311](https://github.com/microsoft/vscode/issues/314311)); Cursor, Gemini emit it too. Git's own `SubmittingPatches` defines it narrowly: *"used to indicate that people exchanged drafts of a patch before submitting it."* ([git SubmittingPatches](https://git-scm.com/docs/SubmittingPatches)) | Yes | **Yes** — the only token GitHub renders as a co-author | **Strongest / "equal credit."** Explicitly asserts *authorship*. Directly the thing USCO says not to do ("list an AI … as an author or co-author"). ([USCO](https://www.copyright.gov/ai/ai_policy_guidance.pdf)) | **Weakest.** Widely read as a legal-authorship claim; several projects ban it for AI (Kubernetes, Mesa, attrs, pip, Requests — see §2). Does not itself add a `Signed-off-by`, so no DCO forgery, but the *authorship implication* is the problem. | Human-to-human collaboration; solo dev who wants zero-config tool default and doesn't care about the connotation. **Avoid for AI in DCO/CLA projects.** |
| **`Assisted-by:`** | **The convergent open-source standard for AI-as-tool.** Prescribed by the **merged Linux kernel** policy: `Assisted-by: AGENT_NAME:MODEL_VERSION [TOOL1] [TOOL2]`, e.g. `Assisted-by: Claude:claude-3-opus coccinelle sparse` ([kernel.org coding-assistants](https://kernel.org/doc/html/next/process/coding-assistants.html)); **OpenInfra** (minor AI help) ([OpenInfra AI policy](https://openinfra.org/legal/ai-policy/)); **IREE** ([IREE contributing](https://iree.dev/developers/general/contributing/#ai-tool-use)); **Mesa**, **QGIS**, **STAC**. Proposed for VS Code Copilot as a replacement for `Co-authored-by` ([VS Code #313962](https://github.com/microsoft/vscode/issues/313962), [#314311 outcome](https://github.com/microsoft/vscode/issues/314311)). | Yes | No (inert log text) | **"Mere tool / helper."** Mirrors git's existing `Acked-by`/`Reviewed-by`/`Reported-by` credit trailers; explicitly *not* an authorship claim. Human is author, AI assisted. | **Strong.** Designed to sit alongside a human `Signed-off-by`; AI never signs off (kernel: *"AI agents MUST NOT add Signed-off-by tags. Only humans can legally certify the … DCO"* ([kernel.org](https://kernel.org/doc/html/next/process/coding-assistants.html))). No authorship implication → USCO-safe. | **Default recommendation for AI-permitting OSS with DCO/CLA.** Machine-auditable, foundation-blessed, legally clean. |
| **`Generated-by:`** | **Prescribed by Apache Software Foundation** ("recommend[ed]" not required: include the phrase `Generated-by:` naming the tooling, to feed a future machine-parsable *Tooling-Provenance* file) ([ASF Generative Tooling Guidance](https://www.apache.org/legal/generative-tooling.html)); **OpenInfra** uses `Generated-By:` for *substantial* AI-produced code (vs `Assisted-By` for minor help) ([OpenInfra](https://openinfra.org/legal/ai-policy/)); **Mesa/QGIS/STAC** allow it. Apache Arrow *defers* to ASF's `Generated-by` rather than defining its own. ([Arrow overview](https://arrow.apache.org/docs/dev/developers/overview.html)) | Yes | No | **"Machine-produced / provenance."** Emphasizes the *artifact's origin* (AI wrote it) over crediting a collaborator. Stronger AI-involvement signal than `Assisted-by` but still not an authorship/co-author claim. | **Strong.** Provenance label, not an author line; pairs with human sign-off. USCO-safe (it does not name the AI as *author*, it records that content was AI-generated — arguably closer to the USCO disclosure duty). | Contributions where AI produced *substantial* / whole chunks (vs. autocomplete-level help). Provenance/SBOM tooling. |
| **`Co-developed-by:`** | **Linux kernel *RFC* (July 2025, superseded).** Sasha Levin's first RFC (July 25 2025) used `Co-developed-by: Claude claude-opus-4-20250514` ([LWN 1031473](https://lwn.net/Articles/1031473/)); the merged policy later switched to `Assisted-by` — after suggestions from David Hildenbrand and Konstantin Ryabitsev — because `Co-developed-by`'s required paired `Signed-off-by` conflicts with the no-AI-sign-off rule and "co-developed" over-credits AI as a co-author. ([kernel.org coding-assistants](https://kernel.org/doc/html/next/process/coding-assistants.html), [Phoronix](https://www.phoronix.com/news/Linux-Kernel-AI-Docs-Rules)) In the **Linux kernel's** `submitting-patches` doc, `Co-developed-by` denotes authorship and "**every `Co-developed-by:` must be immediately followed by a `Signed-off-by:` of the associated co-author**." ([kernel submitting-patches](https://www.kernel.org/doc/html/latest/process/submitting-patches.html)) (Note: git's own `SubmittingPatches` does *not* define `Co-developed-by` — this is a kernel convention.) | Yes | No | **Near-`Co-authored-by` — "equal development credit."** Implies the AI *co-developed* the change; the kernel explicitly rejected this framing for AI. | **Weak for AI.** Its kernel semantics *expect* a matching `Signed-off-by` from the named co-developer — which an AI cannot provide — so using it for AI is semantically broken. Kubernetes bans it by name (see §2). | Human co-developers only. **Not recommended for AI** (historical interest / avoid). |
| **`Assisted-By:` / `Generated-By:` (capitalized variants)** | Same tokens as above; casing varies by project (OpenInfra writes `Generated-By:`/`Assisted-By`; kernel writes `Assisted-by:`). Git's style guide: *"Only capitalize the very first letter"* (favor `Signed-off-by` over `Signed-Off-By`). ([git SubmittingPatches](https://git-scm.com/docs/SubmittingPatches)) | Yes | No | Same as base token. | Same as base token. Casing is cosmetic to git (trailer keys aren't case-normalized by git, but tooling usually treats them case-insensitively). | Match the target project's exact casing convention. |

| **`Prompted-by:`** | **Proposal only.** Would credit the *human prompter* or record that output was prompt-driven; no tool/project uses it. Aligns with USCO's framing of prompts as instructions to a tool, but ambiguous subject. | Safe (no authorship claim). | Niche; ambiguity hurts it. |
| **`Tool:`** | **Proposal / ad-hoc.** Fabiorehm's two-trailer convention splits `Coding-Agent: Claude Code` + `Model: claude-opus-4-6`; a generic `Tool:` line is the same idea. **Pattern:** the DCO/foundation world (kernel, Apache, OpenInfra, Mesa, QGIS) has converged on **`Assisted-by` (help) + `Generated-by` (substantial)** and treats `Co-authored-by`/`Co-developed-by` as *authorship claims to avoid*; Kubernetes goes further and rejects trailers entirely for prose disclosure; the coding-*tools* (Claude Code, Copilot, Cursor) still default to `Co-authored-by`, which is precisely the friction driving this whole debate. *Confidence: high on the kernel/Apache/OpenInfra/Kubernetes/IREE rows (direct policy text); medium on the Mesa/attrs/pip/Requests/LinkML/Pytest rows (relayed via the community catalog because the underlying docs are bot-walled or numerous — treat as directional and verify the specific project before quoting).*

## 3. Distilled guidance for the recommendation matrix

- **Structured + legally clean + interoperable → `Assisted-by:`** (light AI help) escalating to **`Generated-by:`** (substantial AI output). This is the foundation-blessed pair and the safe default for OSS with DCO/CLA. ([kernel.org](https://kernel.org/doc/html/next/process/coding-assistants.html), [openinfra.org](https://openinfra.org/legal/ai-policy/), [apache.org](https://www.apache.org/legal/generative-tooling.html))
- **`Co-authored-by:`** is the *only* token that (a) makes GitHub render a co-author chip and (b) asserts authorship — great for human collaborators, but it's the wording USCO warns against for AI and that DCO projects increasingly ban. Its ubiquity is a *tooling default*, not a considered standard. ([USCO](https://www.copyright.gov/ai/ai_policy_guidance.pdf), [Claude Code #66602](https://github.com/anthropics/claude-code/issues/66602))
- **`Co-developed-by:`** is semantically broken for AI (its *kernel* meaning expects a matching human `Signed-off-by` from the named co-developer, which an AI cannot provide) — avoid.
- **Plain body-text note** is the right call *only* where the project bans trailers (Kubernetes) — it's flexible and authorship-neutral but not machine-auditable.
- **`Aided-by` / `Generated-with` / `Generated-using` / `Partly-generated-with` / `Co-developed-with` / `Written-with` / `Prompted-by` / `Tool:`** are all valid trailers but **proposal-only** with no adoption; choosing them sacrifices interoperability for no connotational gain over the standard `Assisted-by`/`Generated-by` pair (except `Tool:`/`Model:` as *supplementary* machine metadata).
